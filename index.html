Модуль 2.1

Linux для Production

Цель — понимать Linux глазами инженера эксплуатации: где лежит конфигурация, как устроены права, какие процессы запущены и что происходит с сетью.

Файловая система

/etc, /var, /opt, /home, /proc, /sys

Пользователи

useradd, usermod, sudo, groups

Права

chmod, chown, umask

Процессы и сеть

ps, top, ss, ip, ufw

Практика

Поднять Ubuntu VPS, отключить root login, настроить SSH keys, fail2ban, ufw и automatic updates.

Модуль 2.2

Systemd

Запустите приложение без Docker, как управляемый сервис с автоматическим рестартом и логами в journalctl.

fastapi.service
[Unit]
Description=FastAPI App
After=network.target

[Service]
WorkingDirectory=/opt/fastapi-app
ExecStart=/usr/bin/python main.py
Restart=always
RestartSec=5
User=app

[Install]
WantedBy=multi-user.target

Учебный маршрут

Как проходить Production Deployment

Этап лучше проходить как инженерный спринт: сначала фиксируете исходное состояние локального приложения, затем постепенно добавляете эксплуатационные слои и документируете каждую команду в runbook.

Неделя 1 · VPS и Linux baseline

Поднимите чистую Ubuntu, создайте отдельного пользователя, запретите root login, включите firewall, fail2ban и unattended upgrades. Результат — защищённый сервер, к которому можно подключаться только по SSH key.

Неделя 2 · systemd и bare-metal запуск

Запустите FastAPI без Docker, опишите unit-файл, проверьте restart policy, логи через journalctl и корректное завершение процесса по SIGTERM.

Неделя 3 · Docker stack

Перенесите приложение в Docker Compose, добавьте PostgreSQL, Redis, healthcheck, named volumes, isolated networks и ограничения ресурсов.

Неделя 4 · Nginx, HTTPS и база

Подключите домен, TLS, security headers, rate limiting, static files, backup PostgreSQL и проверку восстановления на отдельной базе.

Ключевой артефакт

Для каждого шага сохраняйте runbook: команды, конфиги, ожидаемый результат, способ проверки и план отката. Это превращает практику в reusable production knowledge.
